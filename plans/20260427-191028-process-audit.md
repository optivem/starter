## Missing branches / gaps — NOT auto-applied

### 1. Asymmetric "External System Driver Interface Changed?" diamond in the AT cycle

**Where:** `docs/atdd/process/cycles.md:18-22`

**What is missing:** The AT cycle diagram shows the **Yes** branch of `External System Driver Interface Changed?` (→ Contract Test Sub-Process, then continue) but the **No** branch is implicit — control falls through to the next decision (`System Driver Interface Changed?`) without an explicit `No` arrow. Compare with the same diamond inside the CT sub-process at `cycles.md:54`, which explicitly draws **No → CT - GREEN - STUB**. Readers parsing the AT diagram have to infer the No path; readers parsing the CT diagram do not.

**Question for the user:** Add an explicit `No` arrow on the AT-cycle diamond (and update `orchestrator-diagram.md` to match), or document a convention that "unlabelled fallthrough = No" once at the top of the doc?

### 2. Scenario Loop termination is undefined

**Where:** `docs/atdd/process/cycles.md:68-74`

**What is missing:** The Scenario Loop says "Continue until all scenarios are GREEN" but does not define:
- What happens at termination — does the orchestrator exit silently, post a final issue comment, run a smoke/regression suite, hand off to release-agent?
- What to do when a scenario cannot be made green (legacy bug, blocked by external dep, deferred). Is there a deferral path, or does the orchestrator stall?
- Whether re-entering the loop after termination (re-running the orchestrator on the same ticket) is idempotent or an error.

**Question for the user:** Define the termination behaviour and the deferral path explicitly, or confirm these are intentionally out of scope and belong in a separate doc.

### 3. CT sub-process does not re-check `shop/` driver changes on return

**Where:** `docs/atdd/process/cycles.md:64` ("After the contract test sub-process completes, return to the AT cycle and continue with the system driver check.")

**What is missing:** The flow returns to the AT cycle and continues at the `System Driver Interface Changed?` check based on flags set during `AT - RED - DSL`. But the CT sub-process itself can extend DSL interfaces (`CT - RED - DSL` step 3: "Update the Driver interfaces as needed"), which could in principle add `shop/` driver methods too. The doc treats the original AT flag as authoritative on return; whether CT-induced `shop/` changes can occur — and if so, whether the flag must be re-evaluated — is not stated.

**Question for the user:** Is it a hard rule that CT - RED - DSL only touches `external/` drivers (and never `shop/`)? If yes, state that constraint explicitly in `contract-tests.md`. If no, define how the AT flag is re-evaluated on return.

### 4. STOP-after-COMMIT semantics not reflected in the phase-to-agent table

**Where:** `docs/atdd/process/acceptance-tests.md:56-57` (and equivalent CT phases) vs `docs/atdd/process/cycles.md:80-89`

**What is missing:** The COMMIT phases end with "STOP. Do not proceed further. Phase progression is controlled by the orchestrator, not by this agent." The Phase-to-Agent Mapping table describes COMMIT as "commit + push" with no mention of STOP. A reader of the table alone would not know that COMMIT phases stop and yield control back to the orchestrator.

**Question for the user:** Add a "STOP after commit" column or note to the Phase-to-Agent Mapping table, or merge the STOP semantics into the table's existing "Notes" column?

## Needs-decision — design choices, not bugs (NOT auto-applied)

### 1. Non-recursion rule for CT documented in only one place

**Observation:** `docs/atdd/process/contract-tests.md:43` explicitly disables recursive triggering: "No recursive triggering — this flag is used only for the CT cycle's own skip logic." The symmetric question (can `AT - RED - DSL` recurse via repeated CT sub-processes within a single AT cycle?) is not addressed in `acceptance-tests.md`.

**Canonical / alternative practice:** In canonical ATDD, the outer AT loop is non-recursive; CDC/contract tests fire once per scenario where external dependencies change. Stating the rule in only one direction risks ambiguity if the AT cycle is later edited.

**Question for the user:** Hoist the non-recursion principle into a shared section (e.g. `cycles.md` or the top of `acceptance-tests.md`) so both cycles reference one source of truth, or leave it where it is because CT is the only place recursion is even plausible?
