### 1. CT - GREEN - STUBS has no dedicated agent file — who runs it?

**Canonical / alternative practice:** Two reasonable options: (a) create a new `atdd-stub` agent that `@include`s `ct-green-stubs.md` and `shared-commit-confirmation.md`; (b) extend `atdd-driver` to also handle CT - GREEN - STUBS (it already loads `ct-red-external-driver.md`, so adding `ct-green-stubs.md` is one `@include` line); (c) extend `atdd-test` (since the stub phase enables tests and runs them, similar to test concerns).

**Question for the user:** Create `atdd-stub`, fold into `atdd-driver`, or fold into `atdd-test`? The first matches the per-phase-agent pattern; the second collapses external-driver and stub into one agent (they're sequential phases with similar stop behaviour); the third treats stubs as test infrastructure.

### 2. `atdd-driver.md` describes phase names that don't match the per-phase docs

**Observation:** `atdd-driver.md` lines 19–20 list its accepted phases as `AT - RED - DRIVER - WRITE/COMMIT` and `CT - RED - DRIVER - WRITE/COMMIT`. The per-phase docs name them `AT - RED - SYSTEM DRIVER` (file `at-red-system-driver.md`) and `CT - RED - EXTERNAL DRIVER` (file `ct-red-external-driver.md`). The agent's `@include` lines (12–13) reference the correct file names; only the prose phase labels diverge.

**Canonical / alternative practice:** The phase names in the per-phase docs are correct (they distinguish system from external drivers). The agent body abbreviates both to `DRIVER`, presumably because one agent handles both.

**Question for the user:** Was the abbreviation deliberate (one agent, doesn't care about system vs external) or accidental? If deliberate, document the convention in `at-cycle-conventions.md` / `ct-cycle-conventions.md` ("the canonical name is `AT - RED - SYSTEM DRIVER`; the agent's input form is `AT - RED - DRIVER` and the agent disambiguates by reading which file the phase belongs to"). If accidental, fix the agent body to use the canonical phase names.

### 3. Three vocabularies for the same three task subtypes

**Observation:** Same three concepts named three different ways: ticket-type form (`system-api-task`), dispatcher-input form (`system-api-redesign`), commit-message form (`SYSTEM API REDESIGN`). All three appear in the docs/agents and all three are needed (commit messages have a strict format, dispatcher labels match the GitHub-label families, cycle routing reads ticket types).

**Canonical / alternative practice:** Two options: (a) add a single mapping table in one canonical place (`glossary.md` or `cycles.md`) and reference it everywhere; (b) collapse to one form everywhere, requiring renames in `atdd-dispatcher.md` task-label families and possibly the GitHub label families themselves.

**Question for the user:** Add a mapping table (low cost), or collapse to one vocabulary (higher cost, larger blast radius)? Process-rule-change item #4 above proposes the table; the collapse would need its own plan.

### 4. Unit-level TDD positioning vs canonical double-loop TDD

**Observation:** The documented cycle is acceptance-test-driven outside-in (`AT - RED - TEST → AT - RED - DSL → AT - RED - SYSTEM DRIVER → AT - GREEN - SYSTEM`), with no inner unit-test loop named anywhere. AT - GREEN - SYSTEM is one phase covering "implement backend + frontend" with no per-class TDD steps.

**Canonical / alternative practice:** The Farley/Adzic double-loop TDD prescription has an *inner* loop of TDD on units inside each AT GREEN step. The docs here don't call this out — possibly because the agents handle unit TDD as inside-the-agent detail, possibly because it's omitted by design.

**Question for the user:** Is unit-level TDD a deliberate omission (agent-internal concern, not a process doc concern), or a gap to be filled? If deliberate, add a one-line note in `at-green-system.md` saying so; if a gap, define an inner loop.

### 5. Contract Tests positioning is project-specific, not canonical

**Observation:** `cycles.md:95-123` and the CT per-phase docs position contract tests as a *sub-process triggered by external interface changes* — fired only when AT - RED - DSL adds methods to interfaces under `external/`. Canonical CDC/Pact-style contract testing is run continuously and independently of feature work.

**Question for the user:** Is the project-specific framing intentional (and worth a one-line note in `ct-cycle-conventions.md` explicitly distinguishing it from canonical CDC), or has the team drifted from canon without realising?

### 6. Idempotence of phase re-runs is undefined

**Observation:** The Resume Detection table (`cycles.md:285-294`) tells the orchestrator where to *resume* mid-cycle, but no doc says what happens when a phase is re-invoked after it has already completed. E.g. if AT - RED - DSL has already committed and a user re-invokes `atdd-dsl` for the same ticket, does it no-op, error, or overwrite?

**Question for the user:** Should phases be idempotent (re-run is a no-op when nothing has changed since the last commit), error on re-run, or overwrite? This affects the `atdd-orchestrator` resume logic and every committing agent.
