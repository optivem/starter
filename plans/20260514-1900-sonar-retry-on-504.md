# Retry SonarCloud upload on transient 5xx (commit-stage workflows)

## Context

gh-optivem acceptance run [25865827466](https://github.com/optivem/gh-optivem/actions/runs/25865827466)
failed in 4 of 4 dotnet matrix combos with:

```
HTTP 504 on https://api.sonarcloud.io/analysis/analyses :
{"message": "Endpoint request timed out"}
```

Root cause: transient SonarCloud server-side timeout during the analysis-upload
phase. The failure is invisible in shop's own CI because shop's SonarCloud
project keys are long-lived (incremental analyses), while gh-optivem's
acceptance test scaffolds a fresh project per run (the cold-start /
project-provisioning path is heavier and more timeout-prone).

The retry has to live in shop because the scanner CLI is invoked from shop's
commit-stage workflows. gh-optivem's `internal/shell/sonarcloud.go` talks to a
different host/endpoint (`sonarcloud.io/api/*` for admin) and is not implicated.

## Scope

- **Stacks:** dotnet + java only. TypeScript workflows use
  `SonarSource/sonarqube-scan-action@v7` (a `uses:` step, not bash) — deferred
  until/unless we observe a TS 504.
- **Phase:** only the analysis-upload call. `begin` and `build` left alone.
- **Shape:** inline `until ... done` in each workflow's `run:` block.
- **Backoff:** 3 attempts, linear `attempts*15`s sleep (15s → 30s → 45s).
  Worst-case 90s sleep added per failed job.

## Files changed (4)

### dotnet — wrap only `end`
- `.github/workflows/monolith-dotnet-commit-stage.yml` (was line 131)
- `.github/workflows/multitier-backend-dotnet-commit-stage.yml` (was line 128)

### java — split `build sonar` so only `sonar` retries
- `.github/workflows/monolith-java-commit-stage.yml` (was line 135)
- `.github/workflows/multitier-backend-java-commit-stage.yml` (was line 135)

Gradle handles `build` and `sonar` as independent tasks, so splitting is
behaviourally identical to `./gradlew build sonar` but skips the build cost
on retry.

## Intentionally NOT changed

- **TS commit-stage workflows** (3 files) — use the upstream `SonarSource/sonarqube-scan-action@v7`
  action. Inline retry shape doesn't fit a `uses:` step. Reopen if a TS combo 504s.
- **`system/.../run-sonar.sh`** — local manual helpers, not invoked by CI. A
  dev who hits a transient 504 locally can rerun the script.
- **gh-optivem** — `internal/shell/sonarcloud.go` targets a different endpoint
  (`sonarcloud.io/api/*`); not part of this failure mode.

## Risks

- **Masks brief SonarCloud outages.** Cap of 3 attempts keeps worst-case sleep
  bounded at ~90s per job.
- **Retries on non-retryable errors too** (401 bad token, 400 bad project key
  — scanner exits non-zero on those). Two wasted retries on a config error is
  acceptable vs. parsing stderr for 5xx markers.
- **Visibility only via workflow log.** The `echo "...attempt N/M..."` line is
  the only signal SonarCloud is degrading. Sufficient until/unless retries
  start firing routinely.

## Validation

1. Open a PR with the 4 file edits.
2. Smoke: trigger the gh-optivem acceptance pipeline against this branch:
   ```
   gh workflow run gh-acceptance-stage -f shop-ref=<branch> -R optivem/gh-optivem
   ```
   Expect all 8 dotnet/java matrix combos to pass.
3. Optional negative test on a throwaway branch: point `sonar.host.url` at
   `https://example.invalid` in one workflow; confirm `retrying in 15s` /
   `retrying in 30s` appear before final exit 1.

## Out of scope / follow-ups

- TS retry coverage (open issue and link from the PR if you want it tracked).
- `run-sonar.sh` retry — low value, manual-only.
- 5xx-only predicate instead of "any non-zero" — premature.
